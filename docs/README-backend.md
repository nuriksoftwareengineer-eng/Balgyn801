# Backend (Spring Boot) — структура проекта

Корень приложения: `src/main/java/com/nurba/java/`.

## Слои и папки

| Пакет | Назначение |
|--------|------------|
| `controller/` | HTTP: REST-эндпоинты под префиксом `/api/v1/...`, приём JSON и multipart. |
| `service/`, `service/Impl/` | Бизнес-логика; интерфейсы в `service/`, реализации в `Impl/`. |
| `repositories/` | Spring Data JPA — доступ к БД. |
| `domain/` | Сущности JPA (`Order`, `Product`, `AppUser`, `CdekShipment`, …). |
| `dto/request/`, `dto/responce/` | Тела запросов и ответов API (в т.ч. опечатка `responce` в путях — исторически). |
| `mapper/` | MapStruct: domain/DTO ↔ response. |
| `config/` | Spring: CORS (`WebConfig`), JWT, OpenAPI, MinIO/S3 (`StorageConfig`, `MinioBucketInitializer`), bootstrap-админ. |
| `security/` | Spring Security: `SecurityConfig`, JWT filter, `UserDetailsService`, обработчики 401/403. |
| `exception/` | `RestExceptionHandler`, доменные `NotFoundException`, `BusinessRuleException`. |
| `enums/` | Статусы заказа, роли, тип доставки, статусы СДЭК и т.д. |
| `api/` | Дополнительные «API»-классы (если используются как слой над сервисами — смотреть по файлу). |

Точка входа: `Balgyn801Application.java`.

## Конфигурация

- `src/main/resources/application.properties` — БД, JWT, bootstrap-админ (`app.security.bootstrap-admin-*`), MinIO (`app.storage.*`), лимиты multipart.
- Тесты: `src/test/resources/application-test.properties` (например, отключение MinIO).

## Безопасность (кратко)

- Публично: регистрация/логин, `GET /product/**`, `POST /order`, `POST /custom-design`, Swagger.
- Роль **ADMIN**: товары (создание/удаление), медиа-загрузка, заказы (список/детали), клиенты, кастом-дизайны (чтение), СДЭК, адреса, позиции заказа.

## Хранилище картинок

- Сервис `MediaStorageService`, реализация с AWS SDK v2 (S3-совместимый endpoint, MinIO).
- `POST /api/v1/media/upload` — только ADMIN, публичный URL возвращается в ответе.

## СДЭК (текущее состояние и пример развития)

**Сейчас:** доменная модель и CRUD-прослойка для отправок (`CdekShipment`, репозиторий, `CdekShipmentServiceImpl` с `getById` / `getAll`), админские контроллеры — по сути **учёт записей в своей БД**, без живого API СДЭК.

**Типичный следующий шаг (пример интеграции):**

1. **Учётные данные** — `client_id` / `client_secret` (или токен) СДЭК в env, не в репозитории.
2. **Клиент** — отдельный класс `CdekApiClient` (RestClient/WebClient): базовый URL тест/прод, OAuth или Basic по документации СДЭК.
3. **Сценарии API** (по бизнесу):
   - расчёт тарифа / список ПВЗ (`deliverypoints`) для виджета на чекауте;
   - создание заказа на доставку после оплаты/подтверждения;
   - трекинг по трек-номеру / webhook от СДЭК (если доступен) — обновление `CdekShipment` и при желании `OrderStatus`.
4. **Синхронизация** — при `deliveryType=CDEK` в `Order` создавать черновик отправки или ссылку на расчёт; после ответа СДЭК сохранять `cdek_order_uuid`, трек, статус в `CdekShipment`.
5. **Ошибки и идемпотентность** — повторные запросы, логирование сырого ответа, не дублировать заказы в СДЭК при ретраях.

Итог: бэкенд остаётся источником правды для магазина; СДЭК — внешний сервис, вызываемый из `service/Impl` и по событиям жизненного цикла заказа.

## Оплата (сейчас нет — типовая логика на потом)

**Сейчас:** заказ создаётся через `POST /api/v1/order` без платёжного шага — оплата вне системы или вручную.

**Как обычно делают (пример целевого потока):**

1. **Состояния заказа** — явно разделить «создан / ожидает оплаты / оплачен / отменён» (`OrderStatus` или отдельное поле `paymentStatus`), чтобы не путать с доставкой.
2. **Создание платежа** — после чекаута фронт вызывает что-то вроде `POST /api/v1/payments/intent` (или привязка к существующему `orderId`): бэкенд создаёт платёж у провайдера, возвращает **client secret**, **redirect URL** или **invoice id** — в зависимости от провайдера.
3. **Провайдер** (на выбор под регион и договор):
   - международные: **Stripe**, **PayPal**;
   - СНГ/РФ: **ЮKassa**, **CloudPayments**, **Robokassa** и т.п.;
   - Казахстан: часто **Kaspi** / банк-эквайринг по API банка — смотреть актуальные оферты и документацию.
4. **Подтверждение оплаты** — только доверенный канал:
   - **webhook** от провайдера на `POST /api/v1/payments/webhook/...` (подпись запроса, идемпотентность по `payment_id`);
   - либо серверный **return URL** + доп. запрос статуса к API провайдера (менее надёжно как единственный источник).
5. **Идемпотентность** — повторные webhook и ретраи не должны дважды списать или дважды пометить заказ оплаченным; хранить `providerPaymentId` в БД.
6. **Фронт** — экран «Оплата»: редирект, виджет или QR; после успеха — «Спасибо» и показ номера заказа; при отмене — вернуть в корзину или на заказ.

Итог: деньги и юридически значимый факт оплаты фиксирует **провайдер**; ваш Spring-приложение переводит заказ в «оплачен» только после **проверки** webhook/статуса API, секреты — только на бэкенде.

## Сборка и запуск

- Локально: `./gradlew bootRun` (нужен PostgreSQL; опционально MinIO или `STORAGE_ENABLED=false`).
- Docker: см. корневой `README.md` и `docker-compose.yml`.
